# Research Answers

## 1. How HTTP applications preserve state across multiple request-response cycles

HTTP is a stateless protocol, which means every request is handled independently and the server does not automatically remember previous interactions from the same user. Web applications preserve state by using tools such as cookies, sessions, and tokens. A cookie is small data stored in the browser and sent back to the server with later requests. In many applications, the cookie contains only a session ID, while the real user data is stored securely on the server. This allows the server to remember that a user has already logged in, what actions they performed, or what data belongs to them. Sessions are especially common in Django because they make authentication and user tracking easier. Token-based methods can also be used, especially in APIs, where the client sends a signed token with each request so the server can verify identity without relying on a server-side session. These approaches allow applications to simulate continuity across many request-response cycles even though HTTP itself is stateless. Without these mechanisms, users would have to log in again or restart their work on every new page request.

## 2. Django database migrations to a server-based relational database like MariaDB

Django migrations are the system Django uses to keep the database structure aligned with the models defined in Python code. When working with a server-based relational database such as MariaDB, the developer first installs a compatible database driver and updates the Django settings file with the correct engine, database name, username, password, host, and port. After creating or changing models, the command `python manage.py makemigrations` generates migration files that describe the required schema changes, such as creating a table, adding a field, or altering a constraint. The command `python manage.py migrate` then applies those changes to the MariaDB database in the correct order. Django also keeps a record of which migrations have already been applied by storing that history in the database. This helps prevent duplicate execution and supports consistent deployment across development, testing, and production environments. In practice, migrations make schema management safer, easier to track, and more professional, especially when multiple developers are working on the same project or when the application is deployed to a live server.
